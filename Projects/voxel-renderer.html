Voxel Renderer

Recreating the Minecraft Renderer

This was my second attempt at a voxel renderer using C++ and OpenGL. The first attempt was unorganized, unoptimized and architected poorly, so for this second attempt i thoroughly researched the topic of voxels before hand and proplerly planned how i wished to architect the project.

Solution

In my first system, each block was an individual mesh. Meshes that where hidden by other blocks where culled, but still required thousands of draw calls a frame. This works poorly with the GPU which is optimised for fewer draw calls passing more data in at a time.

This new system bakes the mesh of each chunk at the start of run time instead. All mesh chunks are then rendererd together, reducing the thousands of render calls to more like 32 or 64 depending on the render distance set in the conifguration.

Texture Atlas

Another system implemented in this version is a texture atlas. This means each face of the mesh can be rendererd without needing to take breaks for different voxel types.

To use this, the texture coordinates of each block would be fetched from the texture atlas instance and pushed to the chunk's mesh.

Terrain Generation

The generated terrain uses a perlin noise implementation to span very large distances without the need of manually building a map. I took a similar approach to Minecraft when it came to using perlin noise, where multiple different layers of perlin noise are used on top of each other

This is a very simple usage of perlin noise, and has the issue where a border is formed every 128 chunks, however the generation still expands infinitely, essentially looping the generated terrain there onwards.

If i was to work on this proejct further, the first change i would work on would be the addition of trees. My first attempt had poor excuses for trees, but i would like a system that allows for procedual trees.
